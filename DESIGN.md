## Problem Statement

Building a digital savings, thrift, and cooperative platform targeting West African markets (ajo/esusu tradition). The core pain: group admins like Reghie run multiple WhatsApp-based thrift groups with no persistent record keeping, forcing them to manually track contributions, chase payments, and manage everything in their head.

## Demand Evidence

- One person (Reghie, "Reghie Collections") has 10 groups (~50 people) ready to migrate to the platform
- Another potential customer got upset and requested a refund due to delays in getting access — pre-payment demand signal
- Real but thin: no deep discovery interviews conducted yet; next step is talking to Reghie

## Status Quo

Admins like Reghie run everything manually via WhatsApp groups:
- No persistent record keeping
- Manual tracking of who has paid
- Sending individual reminders to members
- Managing payout schedules by hand
- Admin carries all cognitive load across multiple groups

## Target User & Narrowest Wedge

**The human:** Reghie — runs "Reghie Collections" thrift business. Groups include business owners, coworkers, married women. She's the buyer and the primary user. Members are participants who just need to know when to pay and when they'll receive their payout.

**Narrowest wedge:** Replace Reghie's 50+ weekly manual WhatsApp reminders with automated WhatsApp notifications. Admin enters members by phone number manually (no automated migration in MVP). Free pilot for 2 months while the platform matures, then $10/month SaaS.

**Single feature proof point:** Reghie stops sending manual reminders within 2 weeks.

## Constraints

- Must work alongside WhatsApp, not replace it
- Reghie and her members won't download a new app initially
- Pilot period before monetization
- Small team (1 engineer assumption)

## Premises

1. The biggest pain is the admin's cognitive load, not the members'. (Agreed)
2. Reghie is both buyer and user — the product must make her life dramatically easier. (Agreed)
3. Building on WhatsApp (not against it) is the right wedge — Reghie and members are already there. (Agreed)
4. The 10 groups (50 people) is one customer to prove the wedge, not a market. (Agreed)
5. White-label is a future aspiration, not the day-one differentiator. The differentiator is operational leverage for the admin. (Refined from original premise)

## Recommended Approach

**Approach A: WhatsApp-First Minimal**

Build on WhatsApp using the Twilio API. Admin gets a clean web dashboard to manage groups and contributions. Members receive WhatsApp messages for payment reminders and payout notifications. Members never need to install an app or create an account.

**Core features (Phase 1):**
- Admin: create group, add members by phone number, record contributions, view ledger, trigger reminder broadcasts
- Members: receive WhatsApp messages, no account needed
- System: automated reminders on schedule, ledger persistence, CSV export for admin

**Tech stack:**
- Backend: Flask + SQLite (fastest path to working prototype)
- WhatsApp: Twilio WhatsApp API (sandbox mode for prototype)
- Frontend: Simple admin dashboard (Bootstrap or similar)
- Host: Render or Railway free tier

**Data model (MVP):**
- `Admin` — from Supabase Auth
- `Group(id, admin_id, name, contribution_amount, payout_schedule, current_cycle_number)` — a thrift group
- `Member(id, group_id, phone, name, rotation_order)` — member with rotation order for payout sequence
- `Contribution(id, member_id, amount, date, recorded_by, source)` — contribution, source: 'manual' or 'whatsapp_import'
- `ReminderRule(id, group_id, days_before_payout, message, is_active)` — when and what to remind
- `ReminderState(group_id, current_cycle_number, last_reminder_sent_at)` — tracks reminder state per cycle for idempotency
- `Payout(id, group_id, cycle_number, member_id, amount, payout_date)` — payout history

**Cycle completion:** Auto-detect (all N members paid → advance to cycle+1) AND admin can manually force cycle advance for edge cases (skipped member, dispute, correction).

**Tech stack (updated from eng review):**
- API layer: FastAPI
- Database: Supabase (PostgreSQL)
- Auth: Supabase Auth (magic link by email)
- Hosting: Vercel
- Cron: Vercel Cron Jobs (built-in, every 15 min)
- WhatsApp: Twilio API (sandbox mode for prototype)
- WhatsApp import: Python parser with review UI (regex + confidence scoring)

**What's skipped:**
- Payment processing integration (M-Pesa, Flutterwave)
- Member-facing app or portal
- Member-side verification (admin dashboard only in MVP)
- Dispute/correction flow (Phase 2)
- WhatsApp Business API production approval (apply early, use sandbox for pilot)
